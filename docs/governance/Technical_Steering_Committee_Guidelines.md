# Technical Steering Committee (TSC) Guidelines – NI Open-Source Program

## Executive Summary

This document outlines the structure, responsibilities, and operating norms of the Technical Steering Committees (SteerCos) that participate in NI's Open-Source Program. These guidelines clarify the scope of influence SteerCo members have, how decisions are made, and how this governance structure aligns with broader NI goals, including contributor experience and recertification pathways.

This governance model was inspired in part by community-led models like the Apache, Linux, and Eclipse Software Foundations, but was adapted to suit the needs of a distributed, part-time volunteer contributor base and a recertification-integrated ecosystem.

**Key Highlights:**

- SteerCo members help define direction, feasibility, and structure — but do not implement.
- The Program Manager acts as BDFL and retains final decision authority.
- All governance operates on transparency, traceability, and volunteer respect.
- Contributions approved by SteerCos fuel NI’s recertification-by-contribution program.

## Purpose and Scope

The Technical Steering Committee (TSC or "SteerCo") plays a guiding role in shaping the technical direction of GitHub repositories governed by NI’s Open-Source Program. Members are not expected to code — their role is to provide insight, feasibility validation, and directional votes on whether specific GitHub issues are safe and meaningful for community contribution.

TSC members influence the contributor experience by:
- Helping approve and define contribution-ready GitHub issues.
- Participating in vote-by-discussion decisions to guide roadmap feasibility.
- Supporting a recertification-aligned workflow by labeling contribution scope (e.g., T-shirt sizing).

---

## Membership Expectations

### Commitment
- TSC members are accepted based on technical merit and a commitment to contribute **two hours per week** of focused attention.
- This **attention quota** can be fulfilled via reviews, discussions, or quick input — it is not implementation work.

### Flexibility
- Members may step away and return without penalty.
- Extensions or reactivation is done by re-submitting the application form.

### Transparency
- Availability signals help the program determine how many issues to open and what load the recertification system can handle.
- Accurate availability protects contributors from stalled reviews and supports NI’s goal of a reliable contributor experience.

---

## Scope of Authority

### Advisory Voting
- TSC members cast votes by **providing technical insight** into GitHub discussions.
- If enough consensus forms, the Program Manager may open the issue for community contribution.

### recertification Enablement
- Each approved issue is assigned a **T-shirt size** (S/M/L), which maps directly to rerecertification credit.
- TSC input ensures the issue is technically realistic, well-scoped, and contributor-friendly.

### BDFL Oversight
- NI’s Program Manager (BDFL) may override TSC consensus in rare cases.
- Overrides are logged and justified to maintain transparency.

---

## Meetings and Communication

### Meeting Norms
- Meetings are **optional and agenda-driven**.
- Insight may be shared asynchronously or in real-time.

### Communication Channels
- GitHub Issues and Discussions
- Private Discord threads (for context before meetings)
- One-on-one check-ins with the Program Manager as needed

### Transcription and Credit
- Meetings are transcribed.
- Technical suggestions raised during meetings are credited and time-stamped to preserve attribution.

---

## Public Recognition

- Participation is **private by default**.
- Members may opt in to be listed in:
  - NI leadership briefings
  - User group presentations
  - Open-source community event slides

Any other public-facing use requires explicit permission.

## FAQ

**Q1: Do I have to attend meetings?**  
**A:** No. Meetings are optional. You can fulfill your role asynchronously via GitHub or private chats.

**Q2: Do I need to implement anything?**  
**A:** No. The SteerCo's job is to guide direction, not to code.

**Q3: What happens if I need to step away?**  
**A:** Just let us know. You’ll never be penalized, and you can return at any time.

**Q4: How do I vote on issues?**  
**A:** By giving your technical opinion in GitHub comments or in meetings. Insight = vote.

**Q5: Why is my vote advisory only?**  
**A:** The Program Manager (BDFL) ensures final alignment with NI’s roadmap and contributor experience goals. All overrides are documented transparently.

**Q6: How does this tie into recertification?**  
**A:** Your decisions directly shape which issues earn contributors rerecertification credit. T-shirt sizing defines contribution effort and maps to credit.
