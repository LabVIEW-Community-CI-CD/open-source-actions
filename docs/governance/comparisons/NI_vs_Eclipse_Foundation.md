# Governance Model Comparison: NI Open-Source Program vs. Eclipse Foundation

## Executive Summary

This document presents a transparent comparison between the governance model used in NI’s Open-Source Program and the Eclipse Foundation’s community-based, vendor-neutral structure. While NI follows a BDFL-led model with volunteer-based Technical Steering Committees (SteerCos), Eclipse uses a formal working group and project management structure governed by meritocratic and institutional participation.

This comparison is not a judgment or endorsement of one model over another. Its purpose is to show how NI’s model was shaped in response to different goals, contributor types, and integration needs (e.g., professional certification).

**Key Takeaways:**

- NI’s model is tailored for flexible, part-time contributors and integrated with NI’s certification system.
- Eclipse governance emphasizes open participation, vendor-neutrality, and institutional roles.
- Both models value transparency and community building but serve different operational contexts.

## Overview

The NI Open-Source Program combines centralized leadership (via BDFL) with flexible Technical Steering Committees (SteerCos) that guide roadmap decisions, issue feasibility, and certification relevance. In contrast, the Eclipse Foundation uses a structured, chartered governance model where working groups and project leads make decisions, often involving institutional stakeholders.

This comparison highlights where the two models align or diverge in leadership structure, contributor engagement, project lifecycle, and transparency.

---

## Comparison Table

| **Aspect** | **NI Open-Source Program (BDFL + SteerCo)** | **Eclipse Foundation Model** |
|-----------|---------------------------------------------|-------------------------------|
| **Leadership Structure** | BDFL-led; Program Manager has final decision authority | Project Leadership Committees (PLCs) + Working Groups govern via charters |
| **Decision-Making** | SteerCo provides advisory input; BDFL may override | Formal project charters and bylaws define decision flow |
| **Governance Philosophy** | Lightweight, volunteer-friendly, async-first | Open, vendor-neutral, and chartered governance processes |
| **Path to Influence** | Application-based SteerCo membership | Contribution-based committership and nomination to leadership roles |
| **Certification Integration** | Directly tied to recertification program via GitHub metadata | No certification integration; projects remain independent of credentialing systems |
| **Transparency** | GitHub-native, meeting transcripts, opt-in public recognition | All governance activity recorded and published via Eclipse portals |
| **Contributor Model** | Flexible, attention-based (~2 hrs/week) | Sustained contributor roles tied to merit and activity |
| **Override Policy** | BDFL may override with justification; logged for transparency | No personal override; governed by project bylaws and working group charters |
| **IP / Legal Handling** | Managed by a separate Organizational Steering Committee (OSC) | Centralized IP/legal governance via the Eclipse Foundation EMO and policies |
| **Project Lifecycle** | Driven by SteerCo capacity and contributor availability | Follows Eclipse Development Process (EDP) and structured review milestones |

## FAQ

**Q1: What makes Eclipse’s governance model unique?**  
**A:** Eclipse emphasizes vendor-neutral governance through chartered working groups. Every project is governed by its own processes under the umbrella of a broader legal and technical structure.

**Q2: Why didn’t NI adopt the Eclipse model directly?**  
**A:** NI required a structure that could support part-time professional volunteers and integrate with certification systems. Eclipse’s model assumes long-term contributor engagement and does not natively support credentialing workflows.

**Q3: Does Eclipse have a BDFL equivalent?**  
**A:** No. Eclipse projects are governed by committees, not individuals, and operate within community-approved charters.

**Q4: Is SteerCo membership permanent like Eclipse Committers?**  
**A:** No. SteerCo roles are fluid. Contributors may join, step back, and return based on availability and interest.

**Q5: Can NI evolve toward Eclipse-style governance?**  
**A:** Yes — if project maturity, contributor volume, and institutional partnerships warrant it. The current model is designed to adapt.

## Conclusion

Both NI and Eclipse Foundation governance models promote open collaboration, but they diverge in structure, authority, and contributor expectations. NI’s model is designed for flexibility, certification alignment, and rapid startup. Eclipse’s model is built for maturity, continuity, and open participation.

This comparison is meant to inform contributors and collaborators — not to prescribe or advocate — and demonstrates how NI’s governance decisions were shaped by its specific goals and constraints.
